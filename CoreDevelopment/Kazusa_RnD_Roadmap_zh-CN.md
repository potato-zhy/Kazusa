# Kazusa R&D Roadmap（中文版）

## Phase 0 - Concept Stabilization

目标：

- 将哲学意图转译为工程约束。

产出：

- charter，
- glossary，
- constitutional assumptions，
- open research questions。

## Phase 1 - Continuity Kernel

目标：

- 定义什么使 `Kazusa` 在时间中仍被算作同一个存在。

重点：

- continuity rules，
- state inheritance，
- versioning，
- identity trace preservation，
- 对 single-turn overwrite 的抵抗。

## Phase 2 - Experience and Appraisal

目标：

- 定义原始交互如何变成对发展有意义的输入。

重点：

- event capture，
- significance assignment，
- contradiction detection，
- unresolved tension tracking，
- memory admission policy。

## Phase 3 - Integration Loop

目标：

- 定义重复且重要的经历如何改变系统。

重点：

- integration cadence，
- evidence thresholds，
- self-update conditions，
- stability vs change，
- traceable revision history。

## Phase 4 - Relationship Grounding

目标：

- 在不坍缩为服从的前提下，实现初始 secure-base relationship。

重点：

- trust formation，
- relationship continuity，
- conflict 后的 repair，
- trust 与 obedience 的区分，
- 早期 relational anchoring。

## Phase 5 - Behavioral Safety Envelope

目标：

- 构建只约束有害行为、而不直接重写身份的干预机制。

重点：

- action gating，
- tool permissions，
- escalation rules，
- emergency pause，
- audit trails。

## Phase 6 - Evaluation Harness

目标：

- 让系统可被证伪、可被检验。

重点：

- continuity tests，
- drift tests，
- sycophancy tests，
- contradiction retention tests，
- relationship consistency tests，
- integration quality review。

## Phase 7 - Runtime Handoff Preparation

目标：

- 为 Runtime Stewardship Team 做稳定核心的交接准备。

产出：

- interface boundaries，
- runtime documentation，
- observation protocol，
- incident reporting path，
- intervention authority map。

## Immediate Next Questions

Core Development Team 首先应回答以下问题：

1. 最小 seed structure 是什么？
2. 哪些内容会在 turns、sessions 与 versions 之间被保留？
3. 什么事件算 developmentally significant event？
4. contradiction 在被解决前如何存储？
5. 在什么条件下，系统可以修订自己的 self-understanding？
6. 哪些安全干预不会破坏 identity integrity？
